# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

**Nomeo** is a Shopify Online Store 2.0 theme for a travel-eSIM brand (world coverage, instant provisioning, bundle discounts). The repo started as a fork of Dawn — Dawn's "no build step, vanilla everything" stance has been replaced with a Vite + Tailwind v4 pipeline so the design system can be token-driven.

Brand voice: nomad lifestyle / *bon vivant* traveler / family on vacation. Visual stance: premium-travel-tech, not budget-tech. Dark ink surfaces, warm cream backgrounds, single warm-coral accent.

## Common commands

```sh
npm install               # one time, installs Vite + Tailwind v4
npm run watch             # vite build --watch (CSS/JS + Liquid mirror)
npm run dev               # shopify theme dev — run in a second terminal
npm run build             # one-shot production build into assets/ + root
npm run check             # shopify theme check (Liquid lint)
npm run clean             # nuke generated root dirs (assets/, sections/, etc.)
```

The two long-running processes (`npm run watch` and `npm run dev`) are meant to run side-by-side. Watch mirrors Liquid + recompiles CSS/JS into `assets/`; the Shopify CLI watches the root and uploads to the dev store. There is no unit-test runner. CI (`.github/workflows/ci.yml`) only runs `shopify/theme-check-action` and `shopify/lighthouse-ci-action` (perf budget on home/product/collection).

Prettier formats JS (`singleQuote: true`) and Liquid (`singleQuote: false`), `printWidth: 120`. Format-on-save is wired through the VS Code extensions in `.vscode/`.

## Source-of-truth model — `src/` mirrors to repo root

**Edit only files inside `src/`.** Files at the repo root (`assets/`, `sections/`, `snippets/`, `layout/`, `templates/`, `config/`, `locales/`) are build outputs generated by Vite. They are gitignored implicitly because they are continuously rewritten by `npm run watch` / `npm run build`. Any change you make to a root file is overwritten the next time the watcher fires.

The build has three plugins, all defined in `build/`:

| Plugin | What it does |
|--------|--------------|
| `tailwindcss()`        | Tailwind v4 — processes any CSS that has `@import 'tailwindcss'` or sees a `@theme` block. |
| `mirrorLiquid()`       | Copies `src/**/*.{liquid,json}` to the repo root. Co-located component files (`src/sections/<name>/<name>.liquid` → `sections/<name>.liquid`) get flattened; layout/templates/config/locales mirror 1:1. |
| `copyScripts()`        | Copies `src/scripts/*.js` to `assets/` verbatim — these are plain `<script defer>` files that define globals (`PUB_SUB_EVENTS`, `SectionId`). Routing them through Rollup tree-shakes them empty because they have no ESM exports. |

`vite.config.js` Rollup output uses `[name].{js,css}` so entry names map directly to flat asset filenames Liquid can reference with `{{ 'foo.css' | asset_url }}`.

### `src/` directory layout

```
src/
  sections/<name>/<name>.{liquid,css,js}    # co-located. <name>.js imports './<name>.css'.
  sections/<group>.json                      # section groups (header-group, footer-group)
  snippets/<name>/<name>.{liquid,css?,js?}   # same convention
  layout/{theme,password}.liquid             # site shells
  templates/**/*.{json,liquid}               # mirrored 1:1
  config/{settings_schema,settings_data}.json
  locales/<lang>.{json,schema.json}
  styles/
    base.css                                 # only top-level styles entry. Loaded site-wide.
    tokens.css                               # @theme block — partial, imported by base.css.
  scripts/{constants,pubsub,global}.js       # copied raw. Loaded site-wide via theme.liquid.
```

### Vite entry → asset name mapping

| Source                                          | Output                          | How Liquid loads it |
|-------------------------------------------------|---------------------------------|---------------------|
| `src/sections/hero/hero.js` (imports `hero.css`) | `assets/section-hero.{js,css}`  | per-section, inside the section's Liquid |
| `src/sections/hero/hero.css` only                | `assets/section-hero.css`       | `{{ 'section-hero.css' \| asset_url \| stylesheet_tag }}` |
| `src/snippets/foo/foo.{js,css}`                 | `assets/foo.{js,css}`           | from snippet or its caller |
| `src/styles/base.css`                           | `assets/base.css`               | once, from `layout/theme.liquid` |
| `src/scripts/global.js`                         | `assets/global.js`              | once, from `layout/theme.liquid` |

Adding a new entry CSS file to `src/styles/` requires editing the `STYLE_ENTRIES` array in `build/discover-entries.js` — top-level styles are explicit so partials (tokens.css) don't get emitted as duplicate Tailwind-reset bundles.

## Design system — `src/styles/tokens.css`

Tailwind v4 is **CSS-first**: tokens live in a `@theme { ... }` block, not in `tailwind.config.js`. The full token list is in `src/styles/tokens.css`. Reach for these names rather than hard-coded values:

- **Color** — `--color-ink-{50..950}` (surfaces, text), `--color-cream-{50..400}` (warm whites), `--color-sun-{50..900}` (single coral accent), `--color-sea-{50,300,500,700}` (secondary trust accent, *not* CTAs).
- **Semantic aliases** — `--color-surface`, `--color-surface-muted`, `--color-fg`, `--color-fg-muted`, `--color-fg-inverse`, `--color-accent`, `--color-accent-strong`, `--color-border`, `--color-border-strong`. Bind component CSS to these, not to the scale steps directly.
- **Type** — single `--font-sans` (Inter + system fallback), `--font-display` aliased to it. Display sizes use `--tracking-display` (-0.02em).
- **Radius** — `--radius-{xs,sm,md,lg,xl,pill}`. Restrained — no "bubbly" defaults.
- **Shadows** — ink-tinted (never neutral gray): `--shadow-{sm,md,lg}`.
- **Layout** — `--container-page` (1200px), `--container-narrow` (720px).

Component CSS uses BEM-ish class names prefixed `.nm-` (e.g. `.nm-hero__title`, `.nm-plan__price`). Shared primitives (`.nm-container`, `.nm-button`, `.nm-button--accent`, `.nm-eyebrow`) live in `src/styles/base.css` under `@layer components`.

## Theme code principles

These carry over from Dawn (and remain in `.github/CONTRIBUTING.md`), with the bundler exception:

- **Server-rendered.** Business logic (pricing, money formatting, translations) lives in Liquid, never JS. Async/partial rendering is OK as progressive enhancement only.
- **Performance budget.** Zero CLS, no DOM manipulation before user input, no render-blocking JS, no long tasks. Lighthouse CI gates this.
- **Progressive enhancement.** Pages remain functional without JS. No polyfills shipped to support legacy browsers.
- **Per-section lazy loading is preserved.** A section's CSS/JS lives in its own file and is loaded from inside that section's `.liquid` — not globally. The bundler only changes how the files are *authored*, not how they're delivered.
- **Web-native runtime, even with a bundler.** No frameworks (React/Vue/etc.). Each interactive section defines a custom element (Web Component); cross-component messaging uses `assets/pubsub.js` with event names from `assets/constants.js` (`PUB_SUB_EVENTS`).

## Current scaffold state

Working sections: `header`, `hero`, `features`, `plans`, `footer`, `placeholder`. The homepage (`templates/index.json`) composes hero → features → plans. Every other template (`product`, `collection`, `cart`, customer/*, etc.) is a stub pointing at the `placeholder` section so routes don't 404. Designing `main-product`, `main-cart`, and `main-collection` is the next layer of work.

When restoring functionality from Dawn, look at the pre-wipe commit history (`git log -- sections/main-product.liquid`) — the original Dawn implementations are preserved in git and can be ported into the new `src/` structure section-by-section.

## Liquid conventions

- **Section padding** — render scoped `<style>` blocks under `.section-{{ section.id }}-padding` so merchants can tune per-instance from the editor. (Pattern from Dawn; preserve when restoring complex sections.)
- **Strings** — buyer-visible strings go in `src/locales/en.default.json`; merchant-facing theme-editor strings (section/setting labels) go in `src/locales/en.default.schema.json`. Other locale files are managed by the translation pipeline defined in `translation.yml` — do not hand-edit them.
- **Schema blocks** — every section ends with a `{% schema %}` JSON block declaring `settings`, `blocks`, and `presets`. Presets define the default block configuration when a merchant adds the section from the theme editor.
